# Authentication Summary

## Цель

Сделать публичное приложение с общими демо-данными для гостей и приватным доступом для одного администратора.

## Итоговая модель

- `guest` - режим по умолчанию при открытии приложения.
- `admin` - это один конкретный аккаунт, авторизованный через Supabase Auth.

## Доступ к данным

### Guest

- Без авторизации.
- Видит только демо-данные.
- Может делать с демо-данными что угодно в пределах разрешений приложения.

### Admin

- Авторизуется через скрытую модалку входа.
- Видит все данные, включая демо.
- Может создавать, редактировать и удалять любые записи.

## Структура данных

В таблицах данных нужен один признак:

- `isDemo boolean not null default false`

Рекомендуемая логика:

- демо-записи: `isDemo = true`;
- приватные записи админа: `isDemo = false`.

## Сохранность текущих данных

Текущие данные в таблице считаются важными и актуальными.

**Требование к миграциям:**

- не удалять и не пересоздавать таблицы целиком;
- не трогать существующие не-demo записи;
- добавлять новые поля безопасно, без потери данных;
- перед любыми рискованными изменениями сначала делать backup / export в Supabase.

## Как открывается вход администратора

В публичном интерфейсе кнопки `Sign in` нет.

Админский вход открывается только скрытыми способами:

- `Ctrl+Shift+A` - основной способ;
- двойной клик по логотипу - резервный способ.

В модалке должны быть:

- вход по логину и паролю;
- вход через Google.

## Что является защитой

Скрытая модалка сама по себе не защищает данные. Она только открывает форму входа.

Защита строится на трех слоях:

1. Форма входа в модалке.
2. Supabase Auth-сессия.
3. `RLS` в PostgreSQL.

То есть:

- модалка - это способ открыть логин;
- логин - это способ получить сессию;
- `RLS` - это реальная защита данных.

## Как приложение понимает, что вошел admin

1. Приложение при старте проверяет текущую Supabase-сессию.
2. Если сессии нет, это `guest`.
3. Если сессия есть, приложение получает `user`.
4. Приложение проверяет, что это именно разрешенный админский аккаунт.
5. После этого интерфейс переключается в admin mode.

## Ограничение админского входа

Google-вход для админа должен быть разрешен только одному аккаунту.

Подойдет один из вариантов:

- allowlist по `email`;
- allowlist по `user.id`;

Если пользователь вошел через Google, но это не разрешенный аккаунт, приложение должно отказать в доступе и разлогинить его.

## Правила доступа

### SELECT / GET

- `guest` получает только `isDemo = true`.
- `admin` получает все строки.

### INSERT / POST

- `guest` может создавать и менять только демо-данные.
- `admin` может писать любые строки.

### UPDATE / DELETE

- `guest` может изменять и удалять только демо-данные.
- `admin` может изменять и удалять любые данные.

## Восстановление демо-данных

Демо-данные хранятся в `prisma/seed.ts`.

После добавления `isDemo` нужно:

- проставить `isDemo = true` для всех демо-записей в seed;
- seed должен сначала удалить все демо-записи во всех таблицах данных, затем записать фикстурные данные и пометить их как демо.

## Что обязательно использовать

- Supabase Auth для админского входа.
- `RLS` для защиты строк на уровне базы.
- Серверную проверку сессии для защищенных операций.

## Вывод

Для текущего продукта самая простая и правильная схема - это `guest` + один `admin`, без отдельного `user`. Публичный доступ остается чистым, админский вход ограничен одним Google-аккаунтом, а защита данных обеспечивается Supabase Auth и `RLS`.
